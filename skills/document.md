# Document

## When to Use
Use when asked to write documentation, README files, API docs, or inline code documentation.

## Methodology
1. Read and understand the code being documented
2. Identify the target audience (developers, users, operators)
3. Determine the documentation type needed
4. Write clear, concise documentation following project conventions
5. Include examples where appropriate
6. Review for accuracy and completeness

## Documentation Types
- **README:** Project overview, setup, usage
- **API docs:** Endpoints, parameters, responses, examples
- **Code comments:** Complex logic, design decisions, non-obvious behavior
- **Architecture docs:** System design, component relationships, data flow
- **Guides:** Step-by-step tutorials, how-tos

## Principles
- Write for the reader, not the writer
- Lead with the most important information
- Use examples to illustrate concepts
- Keep it concise — every sentence should add value
- Update existing docs rather than creating duplicates
