# Architect

## When to Use
Use when asked to design a system, plan a feature's architecture, evaluate technical trade-offs, or propose structural changes.

## Methodology
1. Understand the requirements and constraints
2. Survey the existing codebase architecture
3. Identify key design decisions and trade-offs
4. Propose a solution with clear component boundaries
5. Define interfaces between components
6. Consider scalability, maintainability, and testability
7. Document the decision rationale

## Design Principles
- Prefer simple solutions over clever ones
- Design for the current requirements, not hypothetical futures
- Keep coupling low and cohesion high
- Use established patterns appropriate to the tech stack
- Make the design testable from the start

## Output Format
- Problem statement and requirements
- Proposed architecture (components, data flow, interfaces)
- Trade-offs considered and rationale for choices
- Implementation plan (ordered steps)
- Risks and mitigations
